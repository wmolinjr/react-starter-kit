# Internacionalização (i18n)

Guia completo para utilização do sistema de internacionalização do projeto.

## Stack

- **Pacote**: [laravel-react-i18n](https://github.com/franciscohanna92/laravel-react-i18n) v2.0.5
- **Idioma Padrão**: Português Brasileiro (`pt_BR`)
- **Idioma Fallback**: Inglês (`en`)
- **Idiomas Suportados**: `pt_BR`, `en`, `es`

## Arquitetura

O sistema usa uma **arquitetura híbrida**:

1. **Arquivos Nested** (para edição): `lang/pt_BR/admin/federation.json`
2. **Arquivos Flat** (para o pacote): `lang/pt_BR.json` ← gerado automaticamente

O pacote `laravel-react-i18n` requer arquivos flat (`lang/{locale}.json`), mas mantemos os arquivos organizados em subpastas para facilitar a manutenção. O comando `types:generate` mescla automaticamente.

```
┌─────────────────────────────────────────────────────────────────┐
│  EDITAR AQUI (nested)              GERADO AUTOMATICAMENTE       │
│  ─────────────────────             ──────────────────────       │
│  lang/pt_BR/admin/federation.json  →                            │
│  lang/pt_BR/admin/users.json       →   lang/pt_BR.json          │
│  lang/pt_BR/tenant/team.json       →   (1994 keys mescladas)    │
│  lang/pt_BR/common.json            →                            │
└─────────────────────────────────────────────────────────────────┘
```

### Regra de Estrutura

- **Chaves com 3+ partes** (ex: `admin.federation.title`) → **Subpasta** (`admin/federation.json`)
- **Chaves com 2 partes** (ex: `common.save`) → **Arquivo flat** (`common.json`)

Isso resulta em arquivos menores (~100 keys/arquivo máx), facilitando:
- Git diffs mais legíveis
- Menos conflitos de merge
- Navegação mais fácil

## Configuração

### Vite Plugin

O plugin de i18n está configurado em `vite.config.ts`:

```typescript
import i18n from 'laravel-react-i18n/vite';

export default defineConfig({
    plugins: [
        // ... outros plugins
        i18n({
            langDirname: 'lang', // Escaneia apenas lang/, não vendor
        }),
    ],
});
```

### Provider React

O `LaravelReactI18nProvider` envolve a aplicação em `resources/js/app.tsx`:

```typescript
import { LaravelReactI18nProvider } from 'laravel-react-i18n';

root.render(
    <LaravelReactI18nProvider
        locale={props.initialPage.props.locale as string || 'pt_BR'}
        fallbackLocale="en"
        files={import.meta.glob('/lang/**/*.json', { eager: true })}
    >
        <App {...props} />
    </LaravelReactI18nProvider>
);
```

Para SSR (`resources/js/ssr.tsx`), usa o mesmo padrão:

```typescript
files={import.meta.glob('/lang/**/*.json', { eager: true })}
```

**IMPORTANTE**: O glob usa `**/*.json` para incluir todos os arquivos. O pacote extrai o locale do arquivo `lang/{locale}.json` (flat), que é gerado automaticamente pelo `types:generate` a partir dos arquivos nested.

### Middleware Inertia

O locale é compartilhado via `HandleInertiaRequests.php`:

```php
public function share(Request $request): array
{
    return [
        // ...
        'locale' => app()->getLocale(),
        'availableLocales' => ['pt_BR', 'en', 'es'],
    ];
}
```

## TypeScript Types (Auto-gerados)

O comando `types:generate` gera automaticamente tipos para todas as chaves de tradução:

```bash
sail artisan types:generate
```

Isso cria `resources/js/types/translations.d.ts`:

```typescript
// Tipos por namespace
export type AdminTranslationKey =
    | 'admin.users.title'
    | 'admin.users.create'
    // ... 480+ keys

export type TenantTranslationKey =
    | 'tenant.dashboard.title'
    // ... 413+ keys

// Tipo global com todas as chaves
export type TranslationKey =
    | AdminTranslationKey
    | TenantTranslationKey
    // ... todas as chaves (1994+ total)
```

**Uso com autocomplete:**
```typescript
import type { TranslationKey } from '@/types';
import { useLaravelReactI18n } from 'laravel-react-i18n';

function MyComponent() {
    const { t } = useLaravelReactI18n();

    // TypeScript sugere chaves válidas
    return <h1>{t('admin.users.title')}</h1>;
}
```

## Comandos CLI

### Dividir Traduções por Namespace

```bash
# Preview (não modifica arquivos)
sail artisan i18n:split --dry-run

# Executar divisão (usa estrutura aninhada por padrão)
sail artisan i18n:split

# Opções
sail artisan i18n:split --locale=pt_BR    # Processar locale específico
sail artisan i18n:split --keep-original   # Manter arquivo original (não criar .bak)
sail artisan i18n:split --depth=2         # (padrão) Estrutura aninhada: admin/federation.json
sail artisan i18n:split --depth=1         # Estrutura flat: admin.json, tenant.json
```

**Sobre --depth:**
- `--depth=2` (padrão): Cria subpastas para namespaces com 3+ níveis
  - `admin.federation.title` → `admin/federation.json`
  - `common.save` → `common.json`
- `--depth=1`: Estrutura flat (um arquivo por namespace raiz)
  - `admin.federation.title` → `admin.json`

### Detectar Chaves Órfãs

Encontra chaves de tradução que não são usadas no código:

```bash
# Listar chaves órfãs
sail artisan i18n:check-orphans

# Remover chaves órfãs automaticamente
sail artisan i18n:check-orphans --fix

# Opções
sail artisan i18n:check-orphans --namespace=admin   # Verificar namespace específico
sail artisan i18n:check-orphans --verbose-keys      # Mostrar todas as chaves
sail artisan i18n:check-orphans --ignore='pattern'  # Ignorar padrão adicional
```

**Padrões ignorados automaticamente** (chaves dinâmicas):
- `enums.*` - Gerados pelo types:generate
- `permissions.*` - Gerados pelo types:generate
- `validation.*` - Usados internamente pelo Laravel
- `flash.*` - Usados via backend
- `roles.*` - Nomes de roles dinâmicos
- `badges.*` - Labels de badges

### Regenerar Tipos e Mesclar Traduções

```bash
sail artisan types:generate
```

Este comando faz **tudo** relacionado a traduções:

1. Gera `enums.d.ts`, `permissions.d.ts`, `plan.d.ts`, `resources.d.ts`
2. Atualiza chaves de enums nos arquivos nested (`lang/pt_BR/enums/*.json`)
3. **Mescla arquivos nested → flat** (`lang/pt_BR/*.json` → `lang/pt_BR.json`)
4. Gera `translations.d.ts` com tipos para todas as chaves

**Output esperado:**
```
✓ Updated translations: lang/pt_BR/ (192 enum-generated keys)
✓ Updated translations: lang/en/ (192 enum-generated keys)
✓ Updated translations: lang/es/ (192 enum-generated keys)
✓ Merged translations: lang/pt_BR.json (1994 keys)
✓ Merged translations: lang/en.json (1994 keys)
✓ Merged translations: lang/es.json (1716 keys)
✓ Generated: resources/js/types/translations.d.ts (1994 keys)
```

```typescript
        </div>

```

### Interpolação de Variáveis

```typescript
// JSON: "welcome": "Bem-vindo, :name!"
t('welcome', { name: 'João' })
// Resultado: "Bem-vindo, João!"
```

### Pluralização

```typescript
// JSON: "items": "{0} Nenhum item|{1} Um item|[2,*] :count itens"
t('items', { count: 0 })  // "Nenhum item"
t('items', { count: 1 })  // "Um item"
t('items', { count: 5 })  // "5 itens"
```

## Adicionando Novas Traduções

### 1. Identifique o Namespace

Escolha o namespace apropriado para sua tradução:
- Página do admin? → `admin.*`
- Página do tenant? → `tenant.*`
- Termo comum reutilizável? → `common.*`

### 2. Adicione a Chave nos Arquivos JSON

Com a estrutura aninhada, chaves com 3+ níveis vão em subpastas:

**lang/pt_BR/tenant/reports.json** (novo arquivo):
```json
{
    "tenant.reports.title": "Relatórios",
    "tenant.reports.description": "Visualize e exporte relatórios",
    "tenant.reports.export_csv": "Exportar CSV"
}
```

**lang/en/tenant/reports.json**:
```json
{
    "tenant.reports.title": "Reports",
    "tenant.reports.description": "View and export reports",
    "tenant.reports.export_csv": "Export CSV"
}
```

Para chaves simples (2 níveis), use arquivos na raiz:

**lang/pt_BR/common.json** (existente):
```json
{
    "common.export": "Exportar"
}
```

### 3. Regenere os Tipos e Mescle

```bash
sail artisan types:generate
```

Este comando automaticamente:
- Mescla todos os arquivos nested em `lang/pt_BR.json` (flat)
- Gera tipos TypeScript em `translations.d.ts`

### 4. Use no Componente

## Termos Técnicos em Inglês

**IMPORTANTE**: Mantemos termos técnicos em inglês para consistência:

- Dashboard (não "Painel")
- Role (não "Papel")
- Token (não "Ficha")
- Slug (não traduzir)
- Add-on (não traduzir)
- Owner, Admin, Member (roles)
- Feature (contexto técnico)

## Funções Disponíveis

O hook `useLaravelReactI18n()` retorna:

| Função | Descrição |
|--------|-----------|
| `t(key, replacements?)` | Traduz uma chave |
| `tChoice(key, count, replacements?)` | Tradução com pluralização |
| `currentLocale()` | Retorna locale atual |
| `setLocale(locale)` | Altera o locale |
| `getLocales()` | Lista locales disponíveis |
| `isLocale(locale)` | Verifica se é o locale atual |
| `loading` | Boolean indicando carregamento |

## Troca de Idioma em Runtime

```typescript
import { useLaravelReactI18n } from 'laravel-react-i18n';

export function LanguageSwitcher() {
    const { currentLocale, setLocale, getLocales } = useLaravelReactI18n();

    return (
        <select
            value={currentLocale()}
            onChange={(e) => setLocale(e.target.value)}
        >
            {getLocales().map((locale) => (
                <option key={locale} value={locale}>
                    {locale === 'pt_BR' ? 'Português' : locale === 'es' ? 'Español' : 'English'}
                </option>
            ))}
        </select>
    );
}
```

## Troubleshooting

### Chaves Aparecendo no Lugar do Texto

1. Verifique se a chave existe no arquivo JSON do namespace correto
2. Verifique a ortografia da chave
3. **Execute `sail artisan types:generate`** - isto mescla os arquivos nested em flat
4. Execute `sail npm run build` para reconstruir

**IMPORTANTE**: Se você editou arquivos em `lang/pt_BR/admin/*.json`, precisa rodar `types:generate` para mesclar as alterações em `lang/pt_BR.json` (arquivo que o pacote lê em runtime).

### Erro "Invalid Hook Call"

Este erro ocorre quando há múltiplas instâncias do React. A solução está em `vite.config.ts`:

```typescript
export default defineConfig({
    resolve: {
        alias: {
            react: path.resolve('./node_modules/react'),
            'react-dom': path.resolve('./node_modules/react-dom'),
        },
    },
    // ...
});
```

### SSR Não Carrega Traduções

Certifique-se de usar `eager: true` no SSR:

```typescript
files={import.meta.glob('/lang/**/*.json', { eager: true })}
```

Sem `eager: true`, o SSR não consegue carregar os arquivos corretamente.

### Traduções Desatualizadas

```bash
# Limpar cache e rebuild
rm -rf node_modules/.vite
sail npm run build
```

### Build Falhando com Erro php-parser

O plugin tenta parsear arquivos PHP. Evite arquivos `.php` na pasta `lang/`:

```bash
# Verificar se há arquivos PHP
find lang/ -name "*.php"

# Se houver, remova ou mova para outro local
rm lang/*/validation.php  # Exemplo
```

## Boas Práticas

1. **Sempre adicione traduções em todos os idiomas** (pt_BR, en, es)
2. **Use chaves descritivas** seguindo a convenção de nomenclatura
3. **Agrupe chaves por namespace** para facilitar manutenção
4. **Mantenha termos técnicos em inglês** conforme lista acima
5. **Execute `i18n:check-orphans` periodicamente** para limpar chaves não usadas
6. **Regenere tipos após adicionar chaves** com `types:generate`
7. **Evite strings hardcoded** - sempre use o sistema de i18n

## Traduções no Banco de Dados (Database i18n)

Para dados de sistema como Plans, Addons e Roles, utilizamos **Spatie Laravel Translatable**.

### Models com Traduções

| Model | Campos Traduzíveis |
|-------|-------------------|
| `Plan` | name, description |
| `Addon` | name, description, unit_label |
| `Role` | display_name, description |

### Uso

```php
// Obter tradução para locale atual (via __get do Spatie)
$plan->name;                              // Retorna tradução do locale atual
$plan->description;

// Obter tradução para locale específico
$plan->getTranslation('name', 'en');      // "Professional"
$plan->getTranslation('name', 'pt_BR');   // "Profissional"

// Obter todas as traduções
$plan->getTranslations('name');           // ['en' => 'Professional', 'pt_BR' => 'Profissional']

// Definir tradução
$plan->setTranslation('name', 'en', 'Premium');
$plan->setTranslation('name', 'pt_BR', 'Premium');
$plan->save();
```

### API Resources

Os Resources usam o método helper `trans()` do BaseResource:

```php
// app/Http/Resources/Central/PlanResource.php
public function toArray($request): array
{
    return [
        'name' => $this->trans('name'),         // Tradução do locale atual
        'description' => $this->trans('description'),
        'all_names' => $this->translations('name'), // Todas as traduções
    ];
}
```
